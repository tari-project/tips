## Abstract

This TIP proposes two alternative approaches for implementing an ERC20-compatible interface for Tari tokens on Ootle, making it significantly easier for cryptocurrency exchanges to integrate Tari-based assets. By providing familiar APIs and patterns that mirror the widely-adopted ERC20 standard, exchanges can leverage their existing infrastructure and reduce integration complexity. This proposal addresses the current challenge where exchanges can easily add ERC20 tokens but find it difficult to integrate Tari's unique architecture. Two solutions are presented: (A) using fungible resources with standard account components, and (B) creating a dedicated ERC20-like component with state-based balance mappings.

## Motivation

Cryptocurrency exchanges routinely integrate ERC20 tokens on a daily basis using standardized tooling and well-established workflows. However, integrating Tari and tokens built on Tari's Ootle layer presents significant technical barriers due to architectural differences. This disparity creates a competitive disadvantage for Tari adoption.

**Problem Statement:**
- Exchanges have mature infrastructure for ERC20 token integration
- Tari's unique architecture requires custom integration approaches
- Development resources required for Tari integration exceed those for ERC20 tokens
- This friction significantly reduces the likelihood of exchange listings

**Benefits of ERC20-Like Interface:**
- **Reduced Integration Time**: Exchanges can adapt existing ERC20 codebases with minimal changes
- **Lower Development Costs**: Familiar patterns reduce the learning curve
- **Faster Time to Market**: Standard workflows accelerate listing processes
- **Increased Adoption**: Lower barriers encourage more exchanges to list Tari tokens
- **Developer Familiarity**: JavaScript/TypeScript developers can use known patterns

**Target Users:**
- Cryptocurrency exchanges (centralized and decentralized)
- Wallet providers
- Token issuers on Tari
- Integration developers

## Specification

This specification assumes the following context:
1. Exchanges are using revealed (non-confidential) Tari addresses
2. The ERC20-like interface only supports non-confidential funds
3. Exchanges typically use Node.js or similar JavaScript environments
4. Integration requires only a private key and access to an indexer hosting JSON-RPC APIs

### Core Use Cases

Exchanges require support for four primary operations:

1. **UC1 - Wallet Address Generation**: Create unique wallet addresses for each user to receive deposits
2. **UC2 - Event Monitoring**: Query new blocks for transfer events to user addresses, filtered by specific token contracts
3. **UC3 - Account Consolidation**: Sweep funds from multiple user deposit addresses to a single hot wallet
4. **UC4 - Withdrawals**: Send funds from the hot wallet to user-specified addresses

### Common Components

Both proposed solutions utilize:
- **Exchange Node.js Integration**: Adapted version of exchange's existing ERC20 integration code
- **Indexer JSON-RPC API**: RESTful API for blockchain queries
- **Indexer Events WebSocket API**: Real-time event streaming for block updates
- **Tari Account Component**: Standard account for fee payment (and optionally resource storage)
- **Token Contract/Resource**: The specific token being integrated

### Solution A: Fungible Resource with Standard Accounts

This approach leverages existing Ootle primitives without introducing new components.

#### Architecture

Components:
- Exchange integration code (modified)
- Indexer JSON-RPC API (existing)
- **New**: Indexer Events WebSocket API for filtered transfer events
- Tari standard account components (one per deposit address)
- Fungible token resource

#### Implementation Details

**UC1 - Wallet Address Generation:**
- Create a new Tari component (account) for each user
- Component address serves as the unique deposit address
- Requires on-chain transaction and fee payment

**UC2 - Event Monitoring:**
- Implement new WebSocket API on the indexer
- Filter events by resource address (token contract)
- Events include `from` and `to` component addresses
- Alternative: Use existing polling API with increased latency

**UC3 - Account Consolidation:**
- Create multiple transactions to transfer funds from user accounts to hot wallet
- Similar to UTXO-based chains (Bitcoin, Litecoin)
- Each source account requires a separate transaction

**UC4 - Withdrawals:**
- Standard transfer from hot wallet component to destination address
- No special changes required

**API Adapter Layer:**
- Create facades/adapters to map Ootle APIs to `ethers.js`-like interface
- Provide familiar method names and call signatures
- Abstract Ootle-specific details

#### Advantages

1. ✅ No new components or protocols required
2. ✅ Uses battle-tested Ootle primitives
3. ✅ Strong security model from component isolation
4. ✅ Compatible with existing Ootle ecosystem

#### Disadvantages

1. ❌ Account sweeping requires multiple transactions (complexity and higher fees)
2. ❌ Approval mechanisms (like ERC20's `approve`/`transferFrom`) are unclear
3. ❌ Each deposit address requires full component setup with on-chain transaction
4. ❌ Higher on-chain storage costs due to multiple components

### Solution B: ERC20-Like Component with State-Based Balances

This approach introduces a new component type that mimics ERC20's state-based balance model.

#### Architecture

Components:
- Exchange integration code (modified)
- Indexer JSON-RPC API (existing)
- **New**: Indexer Events WebSocket API for component-level events
- Tari account components (for fee payment only)
- **New**: ERC20-like component with balance mapping

#### Implementation Details

**UC1 - Wallet Address Generation:**
- Exchange generates addresses offline using standard key derivation
- No on-chain transaction required
- Significantly reduces setup costs and complexity

**UC2 - Event Monitoring:**
- Implement new WebSocket API on the indexer
- Filter events by component address (the ERC20-like component)
- Events emitted by component include transfer details

**UC3 - Account Consolidation:**
- Use `batch_transfer` method to consolidate multiple accounts in a single transaction
- Dramatically reduces transaction fees compared to Solution A
- Exchange signs each transfer with respective private keys

**UC4 - Withdrawals:**
- Call `transfer` method on the ERC20-like component
- Standard single-transaction withdrawal

**API Adapter Layer:**
- Create adapters to match `ethers.js` Contract interface
- Map method names (e.g., `transfer`, `balanceOf`, `approve`)
- Handle signature and transaction construction

#### Advantages

1. ✅ Offline address generation (no on-chain setup cost)
2. ✅ Efficient batch operations for account sweeping
3. ✅ Native support for ERC20-style approval flows
4. ✅ Potential for custom address and signature types
5. ✅ Lower transaction fees for common operations
6. ✅ More familiar to Ethereum developers

#### Disadvantages

1. ❌ Unclear mapping to native Tari address schemes
2. ❌ Approval flows have been a source of security vulnerabilities in Ethereum
3. ❌ Limited interaction with DEXs and other Ootle contracts
4. ❌ Bypasses many protections provided by Ootle's resource model
5. ❌ Introduces a new component type that must be maintained
6. ❌ State-based model differs from Ootle's resource-oriented approach

## Rationale

### Why ERC20 Compatibility?

The ERC20 token standard is the de facto standard for fungible tokens in the cryptocurrency ecosystem. Most exchanges have:
- Existing codebases optimized for ERC20 integration
- Operational procedures designed around ERC20 patterns
- Staff trained on ERC20 token integration
- Testing frameworks specific to ERC20 interfaces

By providing an ERC20-compatible interface, Tari can leverage this existing infrastructure rather than requiring exchanges to build custom integration layers from scratch.

### Design Decision: Two Solutions

Rather than prescribing a single approach, this TIP presents two solutions with different trade-offs:

**Solution A (Resource-Based)** is more aligned with Ootle's native architecture and security model but requires more complex sweep operations and on-chain setup costs.

**Solution B (State-Based)** provides better compatibility with ERC20 patterns and lower operational costs but introduces architectural divergence from Ootle's resource model.

The community should evaluate both approaches and consider:
- Security implications of each design
- Long-term maintenance burden
- Compatibility with future Ootle features
- Actual adoption likelihood by exchanges

### Alternatives Considered

**1. Custom Tari Integration SDK:**
- More flexible but requires exchanges to adopt entirely new tooling
- Higher barrier to entry
- Rejected due to adoption concerns

**2. Full EVM Compatibility Layer:**
- Would allow direct reuse of ERC20 contracts
- Massive implementation complexity
- Rejected as out of scope for this TIP

**3. Off-Chain Balance Tracking:**
- Exchange tracks balances in their own database
- Periodic settlement to blockchain
- Rejected due to security and transparency concerns

## Backwards Compatibility

Both proposed solutions are additive and do not require changes to existing Ootle components or protocols. However:

### Solution A Compatibility
- ✅ Fully compatible with existing Ootle ecosystem
- ✅ Uses only existing component types
- ✅ No breaking changes

### Solution B Compatibility
- ⚠️ Introduces new component type
- ⚠️ May not interoperate with existing Ootle contracts expecting resource-based tokens
- ⚠️ Tokens on Solution B cannot easily be used in existing Ootle DeFi protocols
- ✅ Existing components unaffected

## Test Cases

Implementation must include comprehensive test coverage for:

### Functional Tests
1. Address generation (Solution B: offline generation)
2. Transfer operations (single and batch)
3. Balance queries
4. Event emission and filtering
5. Error handling (insufficient balance, invalid signatures)

### Integration Tests
1. Full deposit workflow simulation
2. Sweep operations with multiple source addresses
3. Withdrawal operations
4. WebSocket event subscription and delivery
5. Concurrent operations

### Security Tests
1. Signature verification
2. Replay attack prevention
3. Authorization checks
4. Integer overflow/underflow protection (for Solution B)
5. Reentrancy protection (if applicable)

### Performance Tests
1. Gas/fee costs for common operations
2. Batch operation scaling
3. Event query performance
4. WebSocket event delivery latency

### ERC20 Compatibility Tests
1. API signature matching
2. Event format compatibility
3. Error message compatibility
4. Edge case behavior (zero amounts, self-transfers, etc.)
